The oversized-attribute storage technique

A row must fit in 8 KB. Big values move out.

When a row grows past ~2 KB, Postgres compresses large values and, if it still doesn't fit, moves them out-of-line into a hidden per-table TOAST table, leaving an 18-byte pointer inline. A rich car spec sheet crosses that line — so every document gets TOASTed.

TOAST is all-or-nothing per value.

To read one key out of a TOASTed jsonb document, Postgres fetches every chunk of that value from the TOAST table and decompresses the whole thing — even for one 8-character string. And because it lives out-of-line, that fetch is a read from disk: an I/O cost you pay on every row a query inspects. Filtering or sorting on a spec detoasts the documents it touches.

Two backends, one dataset

Store the same cars two ways.

Flip a toggle in the app and run the identical filter against either backend.

Database jsonb

The entire spec sheet in one jsonb column.
  • ›Filters read keys out of the jsonb — detoast per row
  • ›GIN + expression indexes on brand / body_type
  • ✓Any field is queryable in SQL, even deep arrays

Storage hybrid

Filter fields as columns; the blob is a file in a bucket.
  • ✓Filters hit small indexed columns — nothing to detoast
  • ›Full spec fetched from the bucket only on the detail page
  • ✗Fields left in the blob can't be queried in SQL

The whole lesson in one filter

Filter a "deep" spec.

The features array — “360 Camera”, “Heated Seats” — is the tell. In the app it's a multi-select; picks are AND-ed.

Database jsonb

One indexed SQL query via the GIN index (jsonb_path_ops).
-- containment, index-assisted
select * from cars_db
where details @> '{"features":["360 Camera"]}';
-- ~70 ms, no whole-doc download

Storage hybrid

features is inside the blob — not a column. There's no SQL for it.
-- narrow by columns, then...
download 192 JSON files from Storage
scan them in the app
-- hundreds of ms → seconds

You can't WHERE your way into an object-storage file.

Anything you leave in the blob is invisible to SQL. Caching the files only speeds up fetch-by-key; a foreign data wrapper just moves the scan into Postgres without saving the transfer. The fast answer is to keep queryable fields in the database.

Use each for what it's good at

  • ✓jsonb — the payload, plus any field you filter or sort on
  • ✓Storage — large blobs you retrieve by key; cheap bytes, CDN-friendly
  • ›Promote hot fields to columns or STORED generated columns
Local in a minute

Run it yourself.

A Next.js app over a local Supabase stack. Seed fills both tables and uploads the spec files.

supabase start                 # local stack
supabase db reset              # apply the cars schema
npm install
npm run seed                   # 2000 cars: both tables + bucket files

npm run dev                    # http://localhost:3010
npm run bench                  # raw EXPLAIN ANALYZE comparison
